失控的现状

AI 编码 没有护栏

AI 助手很强大，但没有约束。没有治理，它们会擅自改代码、不写文档、把调试代码混入 PR，让代码库陷入混乱。

✕

擅自修改

AI 不经允许就修改文件、重构不该动的代码，把本来能用的代码改坏。

⚠

PR 污染

调试日志、死代码、无关改动悄悄混入 PR，Review 形同虚设。

○

文档腐烂

结构改了文档没更新，一个月后谁也看不懂系统是怎么跑的。

核心原则

七条 铁律

每一条都不可妥协，违反任何一条都视为开发未完成。

01

不能猜测

所有结论必须基于真实命令输出、真实 diff、真实代码上下文。

02

基线分支锁定

开发基线只能是 dev。不得以 main 作为起点。

03

先读文档再动代码

每次开发前必须重新阅读 5 份根文档，不能凭记忆开发。

04

按阶段推进

环境确认 → 对齐基线 → 开发 → 再同步 → 发 PR → 授权后合并，不能跳步。

05

文档必须同步

结构变更、链路变更、规范变更，都要同步更新对应根文档。

06

乱码视为阻塞

中文文档、日志、注释、UI 文案中出现乱码就是开发未完成。

07

最终回执完整

做了什么、是否同步、是否测过、是否有风险，必须说清楚，不能只说「已完成」。

工作流程

7 阶段 PR 管道

每一个 PR 都要经过这套铁律。没有捷径，没有借口 — 只有干净、可审计的交付。

01

环境检查

验证 gh、git 认证、远程配置和当前分支状态，确保环境就绪。

02

同步基线

切换到 dev，拉取最新代码，创建功能分支。绝不从过时的基线起步。

03

开发

先读治理文档再改代码。检查状态流、清理死代码、保持入口文件精简。

04

PR 前同步

git fetch origin 然后 git log --oneline HEAD..origin/dev。基线变了就 rebase。

05

提交与推送

对照基线审查 diff。只包含任务相关改动。无调试代码、无无关格式化。

06

创建 PR

目标分支 dev。结构化正文：改动、风险、测试结果。AI 生成草稿供人工审查。

07

授权合并

仅在明确授权后操作。合并前验证 → 合并 → 合并后校验。分支自动删除。

快速开始

一条命令 部署到你的项目

复制、粘贴，几分钟内让你的 AI 开发工作流具备完整的治理能力。

terminal — zsh
# 远程执行 — 无需克隆
$ curl -sSL https://raw.githubusercontent.com/CoderX42/ai-project-governance/main/scripts/init-governance.sh \
> | bash -s /path/to/your/project --project "MyApp" --branch dev

# 本地执行 — 交互模式
$ bash /path/to/init-governance.sh /path/to/your/project
