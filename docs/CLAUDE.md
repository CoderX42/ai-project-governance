# {{PROJECT_NAME}} - AI 协作总规则

AI 在本仓库进行开发、整理改动、发起 PR、更新 PR、补充说明时，都必须按本文及配套文档执行，不能跳步，不能猜，不能自创流程。

## 开发前必须阅读

任何涉及代码、流程、UI、配置、测试、文档、提交、合并的任务，开始前必须实际阅读或重新核对：

1. [项目文件结构说明.md](./项目文件结构说明.md)
2. [项目完整链路说明.md](./项目完整链路说明.md)
3. [项目开发规范.md](./项目开发规范.md)
4. [AI开发与PR流程.md](./AI开发与PR流程.md)

不能只凭历史记忆、上一次会话摘要或"看起来知道项目"直接开发。

## 核心硬性规则

1. **任何结论都不能猜**，必须基于真实命令输出、真实 diff、真实代码上下文。
2. **开发基线分支只能是 `{{DEFAULT_BRANCH}}`**。AI 不得以 `{{PROD_BRANCH}}` 作为日常开发起点，也不得把 PR 目标分支设为 `{{PROD_BRANCH}}`。
3. **先分析再实现**：如果用户指出"先分析""只分析""暂时不改"，只能输出分析，不得擅自改代码或提交。
4. **按阶段推进**：如果用户要求"开始开发""按照清单开发""提交代码"，必须按阶段推进到完成，不能停在方案层。
5. **文档必须同步**：结构变更、链路变更、规范变更，都要更新对应根文档。
6. **乱码视为阻塞**：中文文档、日志、注释、UI 文案都要做乱码审查，发现乱码视为开发未完成。
7. **最终回执必须完整**：不能只说"已完成"，必须说明改了什么、是否同步文档、是否测过、是否有风险。

## 默认不擅自跑测试/编译

除非规范或任务要求；但结构性改动需要补测试。开发完成后提醒开发者自行验证。

## 最终回执必须包含

1. 实际执行了哪些动作。
2. 当前分支是否已同步最新 `origin/{{DEFAULT_BRANCH}}`。
3. 是否创建/更新了 PR，PR 编号和链接。
4. PR 目标分支是否确认是 `{{DEFAULT_BRANCH}}`。
5. 是否跑了测试；如果没跑，要明确提醒开发者自行测试。
6. 是否已经合并；如果已合并，要说明合并提交和验证结果。
7. 如果有未完成项、未运行测试或残余风险，必须明确说出。

## 一句话执行要求

先确认环境与当前工作区，再对齐最新 `origin/{{DEFAULT_BRANCH}}`，再开发与整理改动，再次同步最新 `origin/{{DEFAULT_BRANCH}}`，最后只向 `{{DEFAULT_BRANCH}}` 发起或更新 PR；只有在开发者明确授权时，才允许把自己的 PR 合并到 `{{DEFAULT_BRANCH}}`。不能跳步，不能猜，不能偷懒。
