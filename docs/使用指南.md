# AI Project Governance — 完整使用指南

> 让 AI 写代码越来越失控？用一套文档 + Skill 模板，把 AI 开发拉回可控轨道。

---

## 一、项目简介

### 1.1 解决什么问题

AI 辅助编程时，常见失控场景：

- AI 擅自改代码、不读文档、猜着写
- PR 混入调试代码、无关改动
- 改完结构不更新文档，一个月后谁也看不懂
- 不同 IDE（Cursor/Claude Code/Trae）规则不互通

**AI Project Governance** 通过一套分层治理规则，把 AI 开发约束成结构化、可审计、可复用的工作流。

### 1.2 核心特性

| 特性 | 说明 |
|------|------|
| 5 份根文档 | 总章程 + 流程规范 + 开发规范 + 链路说明 + 文件结构 |
| 7 个 Cursor Skill | 覆盖初始化 → 需求分析 → 开发 → PR 检查 → 合并验证 |
| 支持 9 个 IDE | Cursor、Claude Code、Trae、Codex、OpenClaw、Gemini CLI 等 |
| 一键初始化 | `bash init-governance.sh /path/to/target` 自动部署到任意项目 |
| 乱码拦截 | 中文文档/注释/UI 文案乱码视为开发阻塞 |

### 1.3 目录结构

```

├── docs/                          # 通用治理规则（IDE 无关）
```

## 二、核心文档说明

### 2.1 五份根文档职责

```
CLAUDE.md              ← AI 每次任务前必须重新阅读的总章程
AI开发与PR流程.md      ← 环境检查 → 开发 → PR → 合并全流程
项目开发规范.md        ← 代码规范、模块边界、测试要求、命名规则
项目完整链路说明.md    ← 业务链路文档（按项目实际补充）
项目文件结构说明.md    ← 文件结构与职责索引（按项目实际补充）
```

### 2.2 核心硬性规则（每一条都重要）

1. **不能猜**：所有结论必须基于真实命令输出、真实 diff、真实代码上下文
2. **开发基线分支只能是 `dev`**：`master`/`main` 只能用于生产发布，AI 不得以它作为日常开发起点
3. **先读文档再动代码**：每次开发前必须重新阅读 5 份根文档
4. **按阶段推进**：环境确认 → 对齐基线 → 开发 → 再同步 → 发 PR → 授权后合并
5. **文档必须同步**：结构/链路/规范变更都要更新对应根文档
6. **乱码视为阻塞**：中文文档、日志、注释、UI 文案乱码 = 开发未完成
7. **最终回执完整**：做了什么、是否同步、是否测过、是否有风险，必须说清楚

### 2.3 占位符说明

根文档中的占位符需要按项目实际替换：

| 占位符 | 示例值 | 说明 |
|--------|--------|------|
| `{{PROJECT_NAME}}` | `MyApp` | 项目名称 |
| `{{DEFAULT_BRANCH}}` | `dev` | 日常开发基线分支 |
| `{{PROD_BRANCH}}` | `main` | 生产分支 |
| `{{TEST_COMMAND}}` | `npm test` | 测试命令 |
| `{{PACKAGE_FILE}}` | `package.json` | 包管理文件 |
| `{{TECH_STACK}}` | `React + Node.js` | 技术栈简述 |

## 三、在 Cursor 中使用（详细案例）

### 3.1 场景一：全新项目，初始化治理

**触发方式**：对 Cursor AI 说以下任意一句

```
"初始化项目 governance"
"set up project rules"
"init governance"
"配置 AI 开发规范"
```

**AI 执行流程**：

1. 检查当前仓库是否已有 5 份根文档
2. 缺失则从模板目录复制（`docs/` 下的 6 份文件 + `CLAUDE.md`）
3. 检查所有 `{{PLACEHOLDER}}` 并提示替换
4. 记录配置到 `.ai-governance.json`

**用户需要做的**：

1. 替换占位符（项目名、分支名、测试命令等）
2. 重新打开 Cursor 或重启对话，让 AI 重新加载 `CLAUDE.md`

---

### 3.2 场景二：新功能开发，任务承接

**触发方式**：对 Cursor AI 说

```
"开始开发新功能：用户下单后自动发送通知"
"帮我分析这个功能"
"start new task"
"analyze this feature"
```

**AI 执行流程**：

1. **读取根文档**：读取 `CLAUDE.md` + 5 份根文档
2. **判断任务类型**：
   - 如果用户说"先分析""只分析"→ 输出分析，**不动代码**
   - 如果用户说"开始开发"→ 输出分阶段开发清单，**进入开发**
3. **需求理解**：如果 `docs/项目需求分析.md` 存在，基于 P0 功能定义理解
4. **输出开发清单**：包含每个阶段的文件列表和自检项

**示例输出**：

```
Action: /apg-intake
Status: complete
Type: implementation
真实目标: 用户下单后自动发送邮件/站内通知
影响范围:
  - 后端: src/services/notification.ts, src/models/order.ts
  - 前端: src/pages/order/OrderDetail.vue
  - 共享: shared/notification-types.ts
  - 文档: docs/项目完整链路说明.md

分阶段开发清单:
  阶段1: 共享定义层 - 新增通知类型枚举和消息结构
  阶段2: 后端实现 - 订单完成触发器 + 消息队列写入
  阶段3: 前端接入 - 订单页通知订阅和展示
  阶段4: 测试覆盖 - 单元测试 + 集成测试
  阶段5: 文档同步 - 更新链路说明和文件结构

Next: 确认后进入开发
```

### 3.3 场景三：PR 前全面检查

**触发方式**：对 Cursor AI 说

```
"帮我检查一下再提交"
"可以提 PR 了"
"check before PR"
"ready to PR"
```

**AI 执行流程**：

1. **Git 卫生检查**：`git status --short`、`git diff --check`
2. **文档同步检查**：确认文件结构说明、链路说明是否已更新
3. **Diff 审查**：`git diff origin/dev...HEAD`
4. **测试与静态检查**：运行 `{{TEST_COMMAND}}`
5. **生成 PR 草稿**：标题 + 正文（本次改动/风险与影响/测试情况）

**示例输出**：

```
Action: /apg-prepr
Status: pass
Git Hygiene: pass
  - 工作区干净，无无关文件
  - 无冲突标记
  - 提交信息："新增订单完成后自动发送通知"
Doc Sync: pass
  - docs/项目文件结构说明.md 已更新
  - docs/项目完整链路说明.md 已更新
Diff: pass
  - 改动范围正确，无泄漏
  - 中文文件无乱码
Test: pass
  - npm test 通过
Next: 可以创建 PR
```

### 3.4 场景四：授权合并 PR

**触发方式**：对 Cursor AI 说

```
"合并这个 PR"
"merge now"
"可以合并了"
```

**AI 执行流程**：

1. **合并前验证**：检查 PR 状态、目标分支、是否落后、是否 draft
2. **执行合并**：`gh pr merge <PR_NUMBER> --merge --delete-branch`
3. **合并后验证**：检查 `state = MERGED`、本地 `dev` 与远端一致

**注意**：AI 只能合并**自己的 PR**，且必须开发者明确授权。

## 五、一键初始化到任意项目

### 5.1 方式一：本地执行（已克隆本仓库）

```bash
# 方式 A：交互模式（引导填写项目信息）
bash /Users/karl/code/ai/ai-project-governance/scripts/init-governance.sh /path/to/your/project

# 方式 B：参数模式（直接指定配置）
bash /Users/karl/code/ai/ai-project-governance/scripts/init-governance.sh \
  --project "MyApp" \
  --branch dev \
  --prod main \
  --test "npm test" \
  --pkg package.json \
  /path/to/your/project
```

### 5.2 方式二：远程执行（推荐）

```bash
curl -sSL https://raw.githubusercontent.com/CoderX42/ai-project-governance/main/scripts/init-governance.sh | bash -s /path/to/your/project --project "MyApp" --branch dev
```

> 首次远程执行时，脚本会自动克隆 governance 仓库到临时目录，用完即删。

## 七、PR 流程详解（全 7 阶段）

```
阶段1: 环境检查
  gh --version && gh auth status && git status --short --branch

阶段2: 同步基线（新任务）
  git switch dev && git pull --ff-only origin dev && git switch -c <feat>

阶段3: 开发（遵守开发规范）
  - 先读文档再改代码
  - 相关逻辑一起检查
  - 清理无用旧代码

阶段4: 发 PR 前再次同步
  git fetch origin && git log --oneline HEAD..origin/dev
  # 如果有新提交 → git rebase origin/dev

阶段5: 提交与推送
  git status --short && git diff origin/dev...HEAD

阶段6: 创建 PR
  gh pr create --base dev --head <feat> --title "..." --body-file <body>

阶段7: 授权后合并（开发者明确授权才执行）
  gh pr merge <PR_NUMBER> --merge --delete-branch
  # 验证 state=MERGED，本地 dev 与 origin/dev 一致
```

## 八、常见问题

### Q: AI 读取不到 CLAUDE.md 怎么办？

确保 `CLAUDE.md` 在项目根目录。如果使用 Cursor，重启对话或新开 Tab 让 AI 重新加载。

### Q: 可以只使用部分文档吗？

可以，但不推荐。5 份文档相互引用，共同构成完整的治理体系。缺少任何一份都可能导致规范不完整。

### Q: 已初始化后发现占位符未替换？

手动编辑 `docs/CLAUDE.md` 和各份 `.md` 文件，将 `{{DEFAULT_BRANCH}}`、`{{PROJECT_NAME}}` 等替换为实际值。

### Q: 支持私有仓库吗？

支持。远程执行时设置 `GOV_REPO` 环境变量：

```bash
GOV_REPO=https://github.com/your-org/ai-project-governance.git \
curl -sSL ... | bash -s /path/to/target
```
