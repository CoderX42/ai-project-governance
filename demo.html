Actions

7 个核心 Action

每个 Action 对应一个检查清单，覆盖完整的协作生命周期。

🗂
/governance-propose
触发词：propose 新功能、创建提案、生成规范文件夹
▼

执行步骤

  1. 解析变更名，中文转 kebab-case（如：添加深色模式 → add-dark-mode）
  2. 在 governance/artifacts/<change-name>/ 下创建完整提案文件夹
  3. 生成 5 个文件：proposal.md、specs/index.md、specs/scenarios.md、design.md、tasks.md
  4. tasks.md 包含 3-5 个 Milestone，每个 Milestone 含检查项，状态标记为 🔄（进行中）/ ✅（完成）/ ⬜（未开始）
  5. 确认目录中有全部 5 个文件后报告完成

输出示例

## Milestone 2: 核心实现 [🔄 2/4]
- [x] Task 2.1: 注册 API 端点
- [x] Task 2.2: 编写数据模型
- [ ] Task 2.3: 实现业务逻辑 ← **建议下一步**
- [ ] Task 2.4: 单元测试
🎯
/governance-help
触发词：下一步是什么、当前进度、governance help、引导我
▼

执行步骤

  1. 扫描 governance/artifacts/*/tasks.md，找出所有进行中（🔄）的变更
  2. 每个进行中的变更输出：变更名、路径、当前 Milestone、完成度
  3. 列出当前 Milestone 中所有未完成任务，第一项标记为 ← 建议下一步
  4. 如无进行中变更，列出所有变更的顶层 Milestone 状态并提示创建新提案

输出格式

## 正在进行：add-dark-mode

路径：`governance/artifacts/add-dark-mode/`

Milestone 2: 核心实现 [🔄 2/4]
- [x] Task 2.1: 注册 API 端点
- [x] Task 2.2: 编写数据模型
- [ ] Task 2.3: 实现业务逻辑 ← **建议下一步**
- [ ] Task 2.4: 单元测试

建议：完成 Task 2.3 后运行 /pre-pr-check 验证
📋
/project-init
触发词：init governance、set up project rules、create CLAUDE.md
▼

执行步骤

  1. 检查当前仓库是否已有 5 份根文档：CLAUDE.md、AI开发与PR流程.md、项目开发规范.md、项目完整链路说明.md、项目文件结构说明.md
  2. 如有缺失，从模板目录复制对应文件
  3. 提示用户替换 {{DEFAULT_BRANCH}}、{{PROD_BRANCH}}、{{TEST_COMMAND}} 等占位符
  4. 记录项目实际值：默认分支、生产分支、测试命令、包管理文件
📥
/task-intake
触发词：start new task、analyze this feature、what should I do next
▼

执行步骤

  1. 读取 CLAUDE.md，如不存在则警告并回退到 /project-init
  2. 读取其他 4 份根文档
  3. 根据用户措辞判断任务类型：分析（analyze/review/explain）→ 只输出分析；实现（implement/develop/fix）→ 进入开发清单
  4. 实现任务输出：需求理解、影响追踪、分阶段开发清单、阶段自检项
🔍
/pre-pr-check
触发词：ready to PR、prepare PR、check before PR
▼

执行步骤

  1. 执行 checklists/pre-pr.md 中的所有检查项
  2. 确认 git status --short 只显示任务相关文件
  3. 确认 git diff --check 无空白错误
  4. 如有结构文件变更，确认 项目文件结构说明.md 已更新
  5. 如有功能流变更，确认 项目完整链路说明.md 已更新
  6. 根据真实 diff 内容生成 PR 标题和正文
✅
/merge-check
触发词：merge now、can I merge、ready to merge
▼

执行步骤

  1. 执行 checklists/merge.md 中的所有检查项
  2. 确认 PR 目标分支是默认开发分支，绝不是生产分支
  3. 确认本地功能分支已吸收最新 origin/dev
  4. 确认 PR 状态为 OPEN 且不是 DRAFT
  5. 执行 gh pr merge 后验证：PR 状态变为 MERGED、本地默认分支快进到合并提交
💡
/requirements-analysis
触发词：new idea、new feature、我有一个想法、需求分析、帮我梳理需求
▼

执行步骤

  1. 检查 docs/项目需求分析.md 是否存在，如有则找出未完成或含 TODO 的章节
  2. 判断是新项目还是现有项目增强，决定是否需要先读 5 份根文档
  3. 分章节引导用户填写：想法概述 → 背景动机 → 用户场景 → 功能优先级 P0/P1/P2 → 非功能需求 → 边界排除 → 成功标准 → 开放问题
  4. P0 功能定义具体验收条件
  5. 完成后交叉检查 项目完整链路说明.md 和 项目文件结构说明.md，冲突记录在第 8 节

文档先行

任何涉及代码、流程、UI、配置、测试的任务，开始前必须实际阅读或重新核对 5 份根文档。不能凭历史记忆直接开发。

核心原则
🚧

阶段门禁

阶段自检未通过时，不能进入下一阶段。每个阶段完成后必须验证：数据流、状态流、日志流、UI 流是否一致。

核心原则
🚫

乱码阻塞

所有中文文档、注释、日志、UI 文案都必须做乱码审查。发现乱码视为开发未完成，不能提交为最终结果。

核心原则

Usage

如何使用

在 Cursor 或 Claude Code 中，通过自然语言触发对应的 Action。

🌱

第一步：初始化项目

对任意新仓库说：init governance 或 set up project rules。AI 会检查 5 份根文档是否齐全，缺失则从模板复制。

Action: /project-init
💡

第二步：想法变提案

有一个新想法？说：propose 新功能：用户头像上传。AI 在 governance/artifacts/ 下生成完整提案文件夹，包含需求、场景、技术方案和任务里程碑。

Action: /governance-propose
🎯

第三步：查看当前进度

随时问：下一步是什么？ 或 当前进度。AI 扫描所有进行中的变更，告诉你当前 Milestone、完成度和建议的下一步操作。

Action: /governance-help
🛠

第四步：开始开发

说：start new task 或 implement xxx。AI 读取根文档、理解需求、追踪影响，输出分阶段开发清单。每个阶段完成后自检再进入下一阶段。

Action: /task-intake
📝

第五步：需求分析

说：我有一个想法 或 需求分析。AI 分章节引导你完善想法、背景、用户场景、P0 功能优先级、验收条件和开放风险。

Action: /requirements-analysis
🚀

第六步：PR 前检查

说：ready to PR 或 check before PR。AI 验证：工作区干净、无无关改动、文档已同步、diff 只含任务相关变更。

Action: /pre-pr-check
🔒

第七步：合并

说：merge now 或 can I merge。AI 验证：PR 状态、目标分支、本地同步，然后执行合并（需明确授权）。

Action: /merge-check
Rules

核心硬性规则

这些规则不可绕过，是 AI 协作的底线。

🚫

Never Guess

所有结论必须基于真实命令输出、真实 diff、真实代码上下文。不能凭记忆、不能靠猜测。

🌿

Dev Only on Dev

开发基线分支只能是 {{DEFAULT_BRANCH}}（如 dev）。不得以 {{PROD_BRANCH}}（如 master）作为日常开发起点，PR 目标分支也绝不能是 master。

📖

Docs Before Code

涉及代码、流程、UI、配置的任务，开始前必须实际阅读根文档。禁止凭"看起来知道项目"直接开发。

🚧

Gate Before Next Stage

阶段自检未通过时，不得进入下一阶段。每个阶段必须完成自检：数据流、状态流、日志流、UI 流是否一致。

🔄

Docs Must Sync

结构变更、链路变更、规范变更，都必须更新对应根文档。不能只改代码不改文档，不能只改标题不改正文细节。

🧹

Garbled Text = Blocked

中文文档、注释、日志、UI 文案出现乱码视为开发未完成。不能提交为最终结果，禁止在未确认编码的情况下批量重写文件。
